# GitHub Issue 作成

## 概要

ユーザーから曖昧な課題の概要を受け取り、コードベース調査とヒアリングを経て、**実装者がユーザーに仕様確認をせずに実装を進められるレベル**のIssueを作成する。

### 前提: ユーザーの特性

- ユーザーは実装の詳細に詳しくない
- ユーザーの要望は曖昧なことが多い（「なんか○○がおかしい」「△△できるようにしたい」程度）
- ユーザーは現在の仕様すら正確には把握していないことがある

このため、Claudeがコードベースと仕様書の知識を活かし、**まず現状を明確にしてユーザーに提示し**、その上でユーザーの意図を実装可能な要件に翻訳する役割を担う。

## 手順

### ステップ1: 課題の概要を受け取る

ユーザーから課題・バグ・改善要望などの概要を受け取る。この時点では曖昧で構わない。

### ステップ2: 現状調査（ユーザーに質問する前に必ず実施）

ユーザーに質問する前に、まずClaudeが自力で以下を調査する。ユーザーに「今どうなっていますか？」と聞いてはいけない。Claudeが調べて提示する。

**調査対象：**

1. **コードベース**: 課題に関連する実装の現在の動作を調査する
   - 関連するファイル、関数、ロジックを特定する
   - 現在どのような処理が行われているか把握する
2. **仕様書（`docs/`）**: 関連する仕様の記載を確認する
   - 該当機能がどう定義されているか確認する
   - 仕様と実装に差異がないかも確認する
3. **既存Issue**: 同様のIssueが既に存在しないか確認する（`gh issue list`）

**調査の深さ**: 課題の範囲に応じて適切に調査する。小さなバグなら関連コード周辺で十分。機能追加なら関連する仕様書も含めて広く調査する。

### ステップ3: 現状提示＋ヒアリング

調査結果をもとに、**現状をユーザーにわかりやすく提示**した上で質問する。

**提示のフォーマット：**

> **現状の動作：**
> 現在は〇〇の画面で△△すると□□になります。
> （例: 「現在はカテゴリを削除すると、そのカテゴリに属していたチャンネルは『未分類』になり、自動ポーリングの対象外になります」）

この「現状の動作」を提示することで、ユーザーが「そうそう、それを変えたい」「いや、そこじゃなくて...」と具体的に答えられるようにする。

**その上で、不足情報を質問する。確認すべき観点：**

- **何を実現したいか**: 「現状は〇〇ですが、これを△△にしたいということですか？」と具体的な選択肢を提示する形で聞く
- **背景・動機**: なぜこの課題が必要か、どんな場面で困っているか
- **具体的な条件**: 数値や閾値が必要な場合は、候補を提示して選んでもらう（「5分・10分・30分のどれがいいですか？」など）
- **エッジケース**: ユーザーが見落としがちなケースをClaudeから提示して確認する（「〇〇の場合はどうしますか？」）

**質問のガイドライン：**

- 技術用語を避け、ユーザーの言葉で質問する
- 「こういう理解で合っていますか？」という確認形式を多用する
- 選択肢や具体例を提示して、ユーザーが「はい/いいえ」や「Aのほう」で答えられるようにする
- 自由記述の質問（「どうしたいですか？」）は最小限にする。ユーザーが答えに困るため
- 一度にまとめて質問する（何往復もしない）。ただし、最初の質問への回答で新たな疑問が出た場合は追加で質問してよい
- Claudeが妥当な判断ができる部分は「〇〇で良いと思いますが、問題ありますか？」とデフォルト提案して確認する

**ヒアリングの完了基準**: 以下がすべて明確になったらヒアリング完了とする。

- 実装者が「ユーザーに確認したいこと」がゼロになっている
- 仕様の曖昧さが残っていない（「たぶんこうだろう」がない）
- エッジケースの扱いが決まっている

### ステップ4: 要件の確定

ヒアリング結果をもとに、要件を整理してユーザーに提示する。

- Issueテンプレート（後述）に沿った形式で要件をまとめる
- 「詳細」セクションには、実装者が判断に迷わない程度の具体性を持たせる
- エッジケースの扱いを明記する
- 完了条件は検証可能な形で書く（「〇〇できること」「〇〇が表示されること」など）
- 影響範囲はステップ2のコードベース調査に基づいて記載する

ユーザーの確認を得たらステップ5へ進む。修正があれば反映して再提示する。

### ステップ5: Issue作成

`gh issue create` コマンドでIssueを作成する。

## Issueテンプレート

```markdown
### 概要
（課題を1〜2文で整理）

### 背景・目的
（なぜこの課題が必要か）

### 詳細
（具体的な要件・仕様。バグの場合は再現手順）

### 完了条件
（検証可能な形式で列挙）
- [ ] 〇〇できること
- [ ] 〇〇が表示されること

### 影響範囲
（関連するファイル・機能。コードベース調査に基づく）

### スコープ外
（今回対応しないこと。必要な場合のみ記載）
```

## タイトルルール

タイトルには種別を示す接頭語を付ける。

| 接頭語 | 用途 |
|---|---|
| `[spec]` | 仕様に関する課題 |
| `[bug]` | バグ・不具合 |
| `[feat]` | 機能追加・改善 |
| `[chore]` | 設定・依存関係・リファクタなど |

例: `[spec] ポーリング間隔の仕様が未定義`

## ラベル運用ルール

Issue 作成時は、課題の種別に応じて以下のラベルを付与する。

| ラベル | 色 | 用途 |
|---|---|---|
| `spec` | 緑 `#0E8A16` | 仕様書に関する課題（矛盾・未定義・不足など詳細は本文に記載） |
| `bug` | — | コードのバグ・不具合 |
| `enhancement` | — | 機能追加・改善 |

その他の GitHub デフォルトラベル（`documentation`, `question` など）は必要に応じて使用する。

### 新しいラベルを作成する場合

- `gh label create "ラベル名" --description "説明" --color "HEXコード"` で作成する
- このセクションに追記して運用ルールを最新に保つ

## 実行コマンド例

```
gh issue create --title "[spec] タイトル" --label "spec" --body "本文"
```
