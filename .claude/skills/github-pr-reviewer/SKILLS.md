# GitHub PR レビュー

## 手順

### フェーズ1: 初回レビュー

1. `gh pr view <番号>` でPRの概要（タイトル・説明・差分ファイル一覧）を取得する
2. `gh pr diff <番号>` で差分全体を取得する
3. 関連するspecファイル（`docs/`）があれば参照し、仕様との整合性を確認する
4. 以下の観点でレビューを行い、指摘事項を整理する（後述「レビュー観点」参照）
5. 指摘事項を日本語でまとめ、`gh pr comment <番号> --body "..."` でPRにコメントする
   - 指摘がない場合も「問題なし」とコメントする
6. **ここで必ずユーザーの指示を待つ**（自動でマージや次のアクションをしない）

### フェーズ2: 再レビュー（「レビュー対応した」と言われたとき）

1. `gh pr diff <番号>` で最新の差分を再取得する
2. 前回コメントした指摘事項が解決されているか一つずつ確認する
3. 新たな問題がないか改めてレビューする
4. 結果をPRにコメントする：
   - 未解決の指摘がある場合: 残っている指摘を列挙してコメント
   - すべて解決済みで品質が十分な場合: 承認コメントをする（後述「承認コメント形式」参照）
5. 未解決の指摘がある場合は**ユーザーの指示を待つ**（フェーズ2を繰り返す）

---

## レビュー観点

### 必須確認項目

- **仕様との整合性**: `docs/` のspecと実装が一致しているか
- **コードの正確性**: バグ・ロジックの誤りがないか
- **セキュリティ**: SQLインジェクション・XSS・認証漏れ等のOWASP Top 10
- **型安全性**: TypeScriptの型が適切か、`any` の不適切な使用がないか
- **APIクォータ**: YouTube APIを呼ぶ実装は `CLAUDE.md` のクォータルールを遵守しているか

### 追加確認項目

- **可読性**: 変数名・関数名が意図を表しているか
- **重複排除**: 既存の処理と重複していないか
- **エラーハンドリング**: 外部APIや非同期処理のエラーが適切に処理されているか
- **Prismaスキーマ**: マイグレーションと `docs/database.md` が一致しているか（DBに関わる変更の場合）

## 指摘コメント形式

```
## PRレビュー結果

### 指摘事項

#### 🔴 必須修正
- `src/foo/bar.ts:42` — （問題の説明と修正方針）

#### 🟡 推奨修正
- `src/foo/baz.ts:10` — （問題の説明と修正方針）

#### 💬 コメント（修正不要）
- （補足・質問・提案）

---

対応が完了したら「レビュー対応した」と教えてください。
```

- 指摘がない場合: 「指摘事項なし。問題は見当たりません。」とだけコメントする

---

## 承認コメント形式

すべての指摘が解決され、マージしてよい品質と判断した場合:

```
## PRレビュー結果（最終）

✅ 全ての指摘事項が解決されました。マージ可能な品質です。

（必要であれば一言コメント）
```

---

## 注意事項

- レビュー後は必ずユーザーの指示を待つ。自動でマージしない
- `gh pr merge` はユーザーから明示的に「マージして」と指示された場合のみ実行する
- 指摘の重大度は「🔴 必須修正」「🟡 推奨修正」「💬 コメント」の3段階で分類する
- 承認コメントは `gh pr comment` で行う（`gh pr review --approve` は使用しない）
