# GitHub Issue 対応

## 手順

1. `gh issue view <番号>` でIssueの内容を取得する
2. 内容を読んで対応方針をユーザーに説明し、確認を取る
3. ブランチを作成する（例: `git checkout -b fix/issue-123`）
4. 実装する
5. テストを実行する（テストが存在する場合。失敗したら修正してから次へ進む）
6. 変更内容をコミットする
7. ブランチをリモートにプッシュする
8. PRを作成する
9. レビューコメントがあれば内容を精査し、対応してコミット・プッシュする
10. PRがマージされたら完了（Issueは自動クローズ）

## 注意事項

- 実装前に必ず対応方針をユーザーに確認する
- ブランチ名はIssue番号を含める
- コミットメッセージにはIssue番号を含める（例: `fix: ログインエラーを修正 #123`）
- テストが存在しない場合は手順5をスキップする
- PR本文には以下を含める:
  - 対応したIssue番号（`Closes #<番号>` 形式でマージ時に自動クローズ）
  - 対応した内容の要約
  - 手順2でユーザーと確認した対応方針（確認した場合）
- `gh issue close` は使用しない（PR自動クローズに統一）

## レビューコメントへの対応方針

レビューコメントを鵜呑みにせず、指摘・推奨修正案の妥当性を以下の観点で精査してから対応を決定する:

- 指摘内容が技術的に正確か（仕様・ライブラリの挙動・ベストプラクティスと照合する）
- 修正案がプロジェクトの設計方針（`docs/`、`CLAUDE.md`）と整合しているか
- 修正によって別の問題が生じないか

**疑問がある場合**はそのまま修正せず、`gh api repos/{owner}/{repo}/issues/{number}/comments` でPRに返信コメントを投稿し、ユーザーに判断を仰ぐ。
