# UI仕様書 - 設定画面

> **スコープ**: 設定画面のレイアウト、コンポーネント、インタラクションの UI 仕様を扱う。ポーリング間隔変更の技術的な反映フローは [architecture.md §6](../architecture.md) を参照。

## 1. 概要

| 項目 | 内容 |
|---|---|
| 画面名 | 設定画面 |
| パス | `/settings` |
| 概要 | ポーリング間隔・コンテンツ保持期間などのアプリ全体設定、チャンネル手動再同期、Web Push通知の有効化・テストを行う |
| 認証 | 要ログイン（未ログイン時は `/login` にリダイレクト） |

## 5. アカウントセクション

```
+------------------------------------------------------------------+
| アカウント                                                         |
| ─────────────────────────────────────────────────────────────── |
| 認証状態      ● 認証済み                                           |
+------------------------------------------------------------------+
```

このセクションは設定画面の**先頭**に配置する。`GET /api/settings` のレスポンスに含まれる `tokenStatus` の値によって表示が変化する。

### 5.1 tokenStatus = "valid"（正常）

```
認証状態      ● 認証済み
```

- 状態バッジ：緑色のドット＋「認証済み」テキスト
- 再認証ボタンは表示しない

### 5.2 tokenStatus = "error"（要再認証）

```
+------------------------------------------------------------------+
| アカウント                                                         |
| ─────────────────────────────────────────────────────────────── |
| 認証状態      ⚠️ 要再認証                                          |
|                                                                  |
| Google アカウントとの連携が切れています。                             |
| 再認証しないとポーリングが動作しません。                               |
|                                                                  |
| [再認証する]                                                       |
+------------------------------------------------------------------+
```

- 状態バッジ：警告アイコン（⚠️）＋「要再認証」テキスト（オレンジまたは赤色）
- 説明テキスト：「Google アカウントとの連携が切れています。再認証しないとポーリングが動作しません。」
- 「再認証する」ボタンを表示する

### 5.3 再認証フロー

「再認証する」ボタンをクリックした際の処理：

```
1. NextAuth の signIn("google") を呼び出す
   （既存の Google OAuth フローをそのまま再利用する）
2. Google の認証画面にリダイレクトされる
3. 認証成功後、NextAuth の signIn コールバックが実行される
   → Account.token_error が NULL にリセットされる
4. 設定画面に戻ると tokenStatus が "valid" になり、バナーが消える
```

- ボタンクリック中はローディング状態にする
- 認証フローはページ遷移（リダイレクト）を伴うため、完了後は設定画面に自動で戻る

## 6. ポーリング設定セクション

```
+------------------------------------------------------------------+
| ポーリング設定                                                      |
| ─────────────────────────────────────────────────────────────── |
| デフォルトポーリング間隔                     [30分 ▼]              |
| カテゴリごとに上書きしない場合に使用されるデフォルト間隔。             |
| 変更は次のポーリングサイクル開始時から反映されます。                    |
|                                                                  |
| 推定1日クォータ使用量: 4,900 / 10,000 units                        |
|                                                                  |
| ⚠️ クォータ警告（条件を満たす場合のみ表示）                           |
|   全カテゴリの合計クォータが警告しきい値を超えています。              |
|   一部のカテゴリのポーリング間隔を長くすることを推奨します。           |
+------------------------------------------------------------------+
```

### 設定項目

| 設定項目 | 説明 | 初期値 |
|---|---|---|
| デフォルトポーリング間隔 | カテゴリごとに個別設定がない場合に使用するデフォルト間隔 | 30分 |

### セレクトボックスの選択肢

| 表示テキスト | 値（分） |
|---|---|
| 5分 | 5 |
| 10分 | 10 |
| 30分（デフォルト） | 30 |
| 1時間 | 60 |

### 挙動

- セレクトボックスを変更すると即座に `PATCH /api/settings` を呼び出す（保存ボタンなし）
- API呼び出し中はセレクトボックスを `disabled` 状態にする
- 成功後：トースト通知で「設定を保存しました」を表示する
- 失敗時：トースト通知でエラーを表示し、元の値に戻す（ロールバック）

### クォータ警告の表示条件

設定画面の初回ロード時に `GET /api/settings` のレスポンスに含まれる以下の値を使って警告を判定する：

| フィールド | 説明 |
|---|---|
| `estimatedDailyQuota` | 全カテゴリ合算の推定1日クォータ消費量（ユニット） |
| `quotaWarningThreshold` | クォータ警告しきい値（ユニット） |
| `quotaDailyLimit` | YouTube API の1日あたりのクォータ上限（ユニット）。使用量表示の分母として使用する |

**警告表示条件:** `estimatedDailyQuota > quotaWarningThreshold`

- 警告メッセージ例：「全カテゴリの合計クォータが警告しきい値を超えています。一部のカテゴリのポーリング間隔を長くすることを推奨します。」
- 推定クォータ使用量は `estimatedDailyQuota / quotaDailyLimit` の形式で表示する（例：「推定1日クォータ使用量: 4,900 / 10,000 units」）

**デフォルトポーリング間隔変更後のクォータ表示更新:**

デフォルトポーリング間隔を変更した場合、PATCH 成功後に `GET /api/settings` を再フェッチし、サーバーが再計算した `estimatedDailyQuota` で警告表示を更新する。フロントエンドはクォータの再計算を行わない。

- フロントエンドは `quotaWarningThreshold`・`quotaDailyLimit`・`estimatedDailyQuota` をハードコードせず、必ず `GET /api/settings` のレスポンス値を参照すること

### クォータ枯渇時の警告バナー

**表示条件:** `GET /api/settings` レスポンスの `quotaExhaustedUntil` が `null` でない場合

```
+------------------------------------------------------------------+
| ⛔ YouTube API クォータが本日分を超過しました。                       |
|    ポーリングは翌日 UTC 00:00 に自動再開します。（日本時間 09:00 頃） |
+------------------------------------------------------------------+
```

- スタイル: shadcn `Alert` の `destructive` バリアント（赤またはオレンジ）
- ポーリング設定セクションの**先頭**（ポーリング間隔セレクトボックスより上）に表示する
- 枯渇中はポーリング間隔の変更操作は引き続き可能（設定値の保存は行われる。次回 TTL 失効後から反映）
- `quotaExhaustedUntil` が `null` に戻る（翌日 UTC 00:00 以降）と、バナーは自動的に消える（`GET /api/settings` の再フェッチによって検知）

**手動ポーリングが枯渇中に実行された場合（`POST /api/categories/{categoryId}/poll` が 503 を返した場合）:**

- トースト通知で「YouTube API クォータが本日分を超過しています。翌日 UTC 00:00 に自動解除されます。」を表示する
- `GET /api/settings` を再フェッチして設定画面のバナーを最新状態に更新する（ダッシュボードからの操作の場合は設定画面には影響しない）

## 8. チャンネル同期セクション

```
+------------------------------------------------------------------+
| チャンネル同期                                                      |
| ─────────────────────────────────────────────────────────────── |
| YouTubeでのチャンネル登録・解除はここから同期できます。                |
| 定期ポーリングではチャンネル一覧は自動更新されません。                  |
|                                                                  |
| [チャンネルを再同期する]                                            |
|                                                                  |
| （完了後のみ表示）                                                  |
| ✅ 追加 2件 / 復元 1件 / 無効化 1件 / 更新 5件                     |
+------------------------------------------------------------------+
```

### 表示要素

| 要素 | 説明 |
|---|---|
| 説明テキスト | 「YouTubeでのチャンネル登録・解除はここから同期できます。定期ポーリングではチャンネル一覧は自動更新されません。」（要件定義 §2.6 より） |
| 「チャンネルを再同期する」ボタン | クリックで手動再同期を実行する |
| 同期結果テキスト | 実行完了後に追加/復元/無効化/更新件数を表示する（直近の実行結果のみ表示。画面を離れると消える） |

### ボタンの状態

| 状態 | 表示 |
|---|---|
| 通常 | 「チャンネルを再同期する」（活性） |
| 実行中 | ボタンを `disabled` にしてスピナーを表示。「同期中...」テキスト |
| 完了（成功） | 通常状態に戻る。ボタン下に結果を表示 |

### 同期結果の表示フォーマット

`追加 {added}件 / 復元 {restored}件 / 無効化 {deactivated}件 / 更新 {updated}件`

- 例：「追加 2件 / 復元 0件 / 無効化 1件 / 更新 5件」
- すべて0件の場合：「変更はありませんでした」
- 実行失敗時：同期結果テキストは表示せず、トースト通知でエラーを表示する

## 9. 通知設定セクション

```
+------------------------------------------------------------------+
| 通知設定                                                           |
| ─────────────────────────────────────────────────────────────── |
| 通知の状態      ● 有効化済み                                        |
|                                                                  |
| [テスト通知を送信する]                                              |
+------------------------------------------------------------------+
```

このセクションの表示はブラウザの通知許可状態によって変わる。

### 9.1 状態の確認ロジック

画面ロード時に以下の順序で通知の状態を確認する：

1. `Notification.permission` の値を確認する
2. `granted`（許可済み）かつ Service Worker サブスクリプション登録済みの場合 → **有効化済み**
3. `granted`（許可済み）かつサブスクリプション未登録の場合 → **許可済み・未登録**（有効化ボタンを表示）
4. `default`（未決定）の場合 → **未有効化**（有効化ボタンを表示）
5. `denied`（拒否済み）の場合 → **拒否済み**（ブラウザ設定の案内を表示）

### 9.2 状態別の表示

#### 有効化済み

```
通知の状態    ● 有効化済み

[テスト通知を送信する]
```

- 状態バッジ：緑色のドット＋「有効化済み」テキスト
- 「テスト通知を送信する」ボタンを表示する

#### 未有効化（許可ダイアログ未表示・または許可済みだがサブスクリプション未登録）

```
通知の状態    ○ 未有効化

ブラウザのWeb Push通知を有効化すると、新着動画・ライブ配信を
リアルタイムで受け取ることができます。

[通知を有効化する]
```

- 状態バッジ：グレーのドット＋「未有効化」テキスト
- 説明テキストを表示する
- 「通知を有効化する」ボタンを表示する

#### 拒否済み

```
通知の状態    ✕ 拒否済み

ブラウザの設定で通知が拒否されています。
ブラウザの設定から通知を許可してください。
```

- 状態バッジ：赤色のバツ＋「拒否済み」テキスト
- 「通知を有効化する」ボタン・「テスト通知を送信する」ボタンは表示しない
- ブラウザ設定への案内テキストのみ表示する

### 9.3 通知の有効化フロー

「通知を有効化する」ボタンをクリックした際の処理：

```
1. Notification.requestPermission() を呼び出す（ブラウザの許可ダイアログを表示）
2. 許可された場合：
   a. navigator.serviceWorker.ready でService Workerの準備を待つ
   b. pushManager.subscribe() で PushSubscription を取得する
      - applicationServerKey: 環境変数 `NEXT_PUBLIC_VAPID_PUBLIC_KEY` の値をそのまま渡す（Base64url 文字列。対象ブラウザはすべて文字列形式を直接受け付ける）
   c. POST /api/notifications/subscriptions で登録する
   d. 成功：状態を「有効化済み」に切り替える
3. 拒否された場合：
   a. 状態を「拒否済み」に切り替える
```

- フロー実行中はボタンをローディング状態にする
- サブスクリプション登録失敗時：トースト通知でエラーを表示し、状態は「未有効化」のままにする

### 9.4 テスト通知送信

「テスト通知を送信する」ボタンをクリックした際の処理：

- `POST /api/notifications/test` を呼び出す
- 呼び出し中はボタンをローディング状態にする（連打防止）
- 成功：トースト通知で「テスト通知を送信しました。デバイスに通知が届いているか確認してください。」を表示する
- 失敗：トースト通知でエラーを表示する

## 10. 各種状態の表示

### 10.1 設定読み込み中

- 各セクションのセレクトボックス部分にスケルトンローダーを表示する

### 10.2 設定の取得失敗

```
設定の取得に失敗しました。ページを再読み込みしてください。
```

- セクション全体にエラーメッセージを表示する

### 10.3 設定変更の成功・失敗

| 操作 | 成功時 | 失敗時 |
|---|---|---|
| ポーリング間隔変更 | トーストで「設定を保存しました」 | トーストでエラーメッセージ（元の値に戻す） |
| コンテンツ保持期間変更 | トーストで「設定を保存しました」 | トーストでエラーメッセージ（元の値に戻す） |
| チャンネル再同期 | ボタン下に件数表示 | トーストでエラーメッセージ |
| テスト通知送信 | トーストで送信完了メッセージ | トーストでエラーメッセージ |
| 再認証 | 認証状態が「認証済み」に更新される（ページリダイレクト後） | Google 認証画面でキャンセルされた場合は設定画面に戻り、tokenStatus が "error" のままになる |

## 12. データ取得・更新フロー

| 操作 | APIコール |
|---|---|
| 初回表示（設定取得・クォータ警告用データ・tokenStatus を含む） | `GET /api/settings` |
| ポーリング間隔変更 | `PATCH /api/settings` `{ pollingIntervalMinutes: <値> }` |
| コンテンツ保持期間変更 | `PATCH /api/settings` `{ contentRetentionDays: <値> }` |
| チャンネル再同期 | `POST /api/settings/sync-channels` |
| 通知サブスクリプション登録 | `POST /api/notifications/subscriptions` |
| テスト通知送信 | `POST /api/notifications/test` |
| 再認証 | NextAuth の `signIn("google")` を呼び出す（API Route ではなく NextAuth フロー） |

- 設定値・`estimatedDailyQuota`・`quotaWarningThreshold`・`quotaDailyLimit`・`tokenStatus`・`quotaExhaustedUntil` はすべて `GET /api/settings` の1回のリクエストで取得する
- TanStack Query でキャッシュし、画面のフォーカス復帰時に再取得する
- ポーリング間隔変更は PATCH 成功後に `GET /api/settings` を再フェッチしてクォータ表示を更新する
- コンテンツ保持期間変更は楽観的更新を適用し、失敗時にロールバックする
- 再認証後にページに戻った際は TanStack Query のキャッシュが invalidate され、`tokenStatus` が最新値に更新される
