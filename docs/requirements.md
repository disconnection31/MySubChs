# 要件定義書 - MySubChs

> **スコープ**: 機能要件・非機能要件・画面一覧・用語定義を扱う。技術的な実装詳細は [architecture.md](./architecture.md)、DB スキーマは [database.md](./database.md) を参照。

## 1. プロジェクト概要

### アプリ名
MySubChs（マイサブチャンネルズ）

### 概要
YouTubeの登録チャンネルを自分好みに分類・管理し、新着動画やライブ配信を効率よく把握するための個人用Webアプリケーション。将来的にTwitchなど他プラットフォームへの対応も見据えた設計とする。

### ターゲットユーザー
自分のみ（シングルユーザー構成）

---

## 2. 機能要件

### 2.1 認証・アカウント連携

- Googleアカウント（OAuth 2.0）でログインできる
- ログイン後、YouTubeの登録チャンネル一覧をYouTube Data API v3経由で自動取得できる
- 取得したチャンネル情報はDBに保存・同期される
- 初回ログイン時にポーリングを1回即時実行する（NextAuthのsignInコールバックでBullMQジョブをエンキュー）。これによりログイン直後からチャンネル一覧・動画一覧が表示される。2回目以降のログインでは即時実行しない（通常のポーリングスケジュールに委ねる）
- **初回ログイン直後のUI**: チャンネル・動画の取得が完了するまでローディング状態を表示する。チャンネルが0件の間は「YouTubeの登録チャンネルを取得中です…」等のメッセージを表示し、取得完了後に一覧を表示する
- ログアウトできる

### 2.2 チャンネル管理

- チャンネルをユーザー定義のカテゴリ（グループ）に分類できる
- 1チャンネルにつきカテゴリは最大1つ（単一カテゴリ所属）
- カテゴリの作成・編集・削除ができる
- カテゴリ名の制約:
  - 空文字不可
  - 最大50文字
  - 同一ユーザー内で一意（重複不可）
- カテゴリの表示順をドラッグ＆ドロップで並び替えできる（`sortOrder` フィールドで管理）
- カテゴリを削除すると、そのカテゴリに所属していたチャンネルは「未分類」扱いになる（チャンネル自体は削除されない）
- チャンネルのカテゴリへの割り当て・解除ができる
- カテゴリに所属していないチャンネルを確認できる（未分類一覧）
- 「未分類」はダッシュボードのサイドバーにユーザー作成カテゴリの下に固定表示される特別カテゴリである。「未分類」を選択すると、どのカテゴリにも属していないチャンネルの動画一覧をメイン画面に表示する
- チャンネルの登録解除（アプリからの除外。OAuthスコープが `youtube.readonly` のため、YouTube側の購読状態は変更しない）はソフトデリートで管理する。解除済みチャンネルはDBに残り `isActive = false` となる
- チャンネル一覧画面でアクティブ／解除済みのフィルタリングができる（デフォルトはアクティブのみ表示）
- **チャンネル一覧画面での再同期案内**: チャンネル一覧画面に「YouTubeでチャンネルの登録・解除を行った場合は、設定画面からチャンネルを再同期してください」という案内と設定画面へのリンクを常時表示する

### 2.3 動画・ライブ一覧

- カテゴリを選択すると、そのカテゴリ内のチャンネルの動画・ライブが一覧表示される
- **表示件数・スクロール**: 無限スクロール方式。1回のリクエストで20件ずつ取得する
- **デフォルトソート**: コンテンツ日時（`contentAt`）の降順（新しい順）
- **ソートトグル**: コンテンツ日時の昇順（古い順）↔ 降順（新しい順）を切り替えるボタンを一覧画面に設置する。昇順にすることで自動削除が近いコンテンツを上位に確認できる
- **ソートキー**: コンテンツ日時（`contentAt`）のみ。チャンネル名・タイトルでのソートは対象外
- 表示内容：
  - 動画タイトル
  - チャンネル名
  - 投稿日時（アーカイブ）または配信開始時刻（ライブ・配信予定）
  - ライブ中 / 配信予定 / アーカイブ の区別がわかるバッジ表示
  - 「後で見る」フラグの管理：
    - 手動でのON/OFFトグル
    - カテゴリ設定に基づく新着コンテンツへの自動フラグ付け
    - 設定された有効期限を過ぎると自動失効する
- **「後で見る」手動OFF の挙動**: 手動でフラグをOFFにしたコンテンツは、以降のポーリングで自動的に再フラグ付けされない。ユーザーが意図的に外した選択を尊重する
- **「後で見る」手動ON の失効**: 手動でフラグをONにした場合、失効日時は設定されない（無期限）。カテゴリの自動失効設定より優先される。ただし**コンテンツ保持期間（§2.7）によるコンテンツ自体の削除は例外なく適用される**
- **CANCELLEDコンテンツの表示**: 配信がキャンセルされたコンテンツ（`status=CANCELLED`）はデフォルトで一覧に表示しない。フィルター「キャンセル済みも表示」をONにした場合のみ表示される
- 「後で見る」のみ表示するフィルタリングができる
- 動画・ライブのサムネイルは表示しない（投稿日時・視聴管理を重視するシンプルな表示）
- 動画タイトルをクリックするとYouTubeの該当ページを新しいタブで開く

### 2.4 通知機能

- 通知ONのカテゴリのチャンネルが以下のアクションを行った際にWeb Push通知を送る：
  - 新しい動画を投稿した
  - ライブ配信を開始した
  - 新しい配信予定（ライブ予約）が登録された（配信予定の通知）
- 配信予定通知はカテゴリ設定の「配信予定の通知」がONの場合のみ送信される（デフォルトOFF）
- 通知はブラウザのWeb Push（PWA）で受け取る
- YouTube APIのポーリング（定期取得）で新着を検知する（初期デフォルトは30分間隔。詳細は §3.2 参照）
- ポーリング間隔は設定画面から変更できる

### 2.5 カテゴリ別設定

**ポーリング設定**

- カテゴリごとに「定期ポーリング」のON/OFFを設定できる（デフォルトON）
- OFFに設定したカテゴリのチャンネルは定期ポーリングの対象から除外される
  - どのカテゴリにも属さない未分類チャンネルは定期ポーリング対象外
- OFFのカテゴリでも「今すぐポーリング」ボタンで手動ポーリングができる
- 手動ポーリングはメイン画面（ダッシュボード）のエリアヘッダーから実行できる
- 手動ポーリング中はボタンをローディング状態にし、完了後にフィードバック（成功/失敗）を表示する
- 手動ポーリングは同一カテゴリに対して**5分間のクールダウン**を設ける（クォータの過剰消費防止）。クールダウン中はボタンを非活性化し残り時間を表示する

**通知設定**

- カテゴリごとに以下の通知のON/OFFを個別に設定できる：
  - 新着動画通知（デフォルトON）
  - ライブ開始通知（デフォルトON）
  - 配信予定の通知（デフォルトOFF）

**後で見る設定**

- カテゴリごとに「後で見る」のデフォルトON/OFFを設定できる
- **新規カテゴリ作成時のデフォルトはOFF**（`watchLaterDefault = false`）
- デフォルトONのカテゴリでは、新着コンテンツに自動で「後で見る」フラグが付く
- カテゴリごとに「後で見る」の自動失効時間を設定できる（1日 / 3日 / 1週間 / 2週間 / 失効なし）（DBは時間単位で保存。1日=24h、3日=72h、1週間=168h、2週間=336h）

### 2.6 設定画面

- ポーリング間隔の設定（5分・10分・30分・1時間）（反映タイミングの詳細は [architecture.md §6](architecture.md) 参照）
- **コンテンツ保持期間の設定**（デフォルト: 2ヶ月）
  - 選択肢: 1ヶ月 / 2ヶ月 / 3ヶ月 / 6ヶ月 / 1年
  - 変更は次回コンテンツクリーンアップジョブ実行時から反映される（即時反映ではない）
- 登録チャンネルの手動再同期: YouTube から現在の登録チャンネル一覧を取得し、DB と同期する（新規追加・復元・無効化・メタデータ更新）。処理フローの詳細は [architecture.md §5「チャンネル同期フロー」](architecture.md) を参照
- **手動再同期ボタンの説明テキスト**: ボタン付近に「YouTubeでのチャンネル登録・解除はここから同期できます。定期ポーリングではチャンネル一覧は自動更新されません。」を表示する
- **手動再同期のフィードバック**: 実行中はボタンをローディング状態にする。完了後に追加・復元・無効化・更新されたチャンネル数を表示する（例：「追加 2件 / 無効化 1件 / 更新 5件」）
- 通知のテスト送信（固定テキスト「通知の設定が完了しています」をWeb Push送信。デバイスの通知許可確認に使用）

### 2.7 コンテンツ自動削除

- 保持期間（§2.6）を超えた `Content` レコードを**物理削除**する（バックグラウンドジョブが毎日1回実行）
- 削除基準日時・削除対象外条件・「後で見る」フラグの扱いの詳細は [architecture.md §6 ContentCleanup](architecture.md) を参照

## 3. 非機能要件

### 3.1 対応環境

- モダンブラウザ（Chrome、Firefox、Safari、Edge の最新版）
- スマートフォンブラウザ対応（レスポンシブデザイン）
- PWA対応（ホーム画面への追加、Web Push通知）

### 3.2 パフォーマンス

- ページ初期表示：3秒以内
- YouTube APIのレートリミット（1日10,000ユニット）を超えないようポーリング設計する。**想定チャンネル数〜100本の場合、`playlistItems.list` (1unit/call) の消費量からポーリング間隔は30分以上を推奨。5分・10分設定時はチャンネル数に応じてUIで警告を表示する（詳細は [architecture.md §6](architecture.md)）**
- エンドポイント別のコストと利用方針の詳細は [ref/youtube-api.md](../ref/youtube-api.md) を参照

### 3.3 セキュリティ

- Google OAuthトークンはサーバーサイドで管理し、クライアントに露出させない
- YouTube APIキーは環境変数で管理する
- 個人利用のため認可（マルチユーザー制御）は不要だが、ログイン必須とする

### 3.4 インフラ・デプロイ

- 初期：Windows PC上のDockerで動作（Docker Compose構成）
- 将来：AWS（ECS Fargate + RDS + ElastiCache）へ移行予定
- コンテナ化によりポータビリティを確保する

## 4. 将来対応（スコープ外・設計考慮のみ）

- Twitch、ニコニコ動画などの他プラットフォームの統合管理
  - プラットフォーム識別子（`platform` フィールド）をDB設計に含めておく
  - チャンネル・動画取得ロジックはプラットフォームごとのアダプターパターンで実装する
- お気に入り動画のブックマーク機能
- AWS移行後のWebhook通知（PubSubHubbub）への切り替え

## 6. 用語定義

| 用語 | 説明 |
|---|---|
| チャンネル | YouTubeのチャンネル（将来は他プラットフォームも含む） |
| カテゴリ | ユーザーが任意に作成するチャンネルの分類グループ |
| コンテンツ | 動画・ライブ配信・配信予定をまとめた総称 |
| ポーリング | 一定間隔でYouTube APIを叩いて新着コンテンツを確認する処理 |
| 後で見る（Watch Later） | コンテンツに付けるフラグ。手動または自動付与。有効期限付きまたは無期限 |
| 自動失効 | 後で見るフラグが有効期限を過ぎると自動的に無効になる仕組み |
| 配信予定の通知 | 新しいライブ予約がYouTubeに登録された際に送信される通知。カテゴリごとにON/OFF設定可（デフォルトOFF） |
| ソフトデリート | レコードを物理削除せず `isActive = false` などのフラグで論理的に無効化する方式 |
| 未分類チャンネル | どのカテゴリにも所属していないチャンネル。削除されたカテゴリからのチャンネルもここに含まれる |
| コンテンツ保持期間 | コンテンツ（動画・ライブ）が自動削除されるまでの期間。設定画面で変更可能（デフォルト2ヶ月）。保持期間を超えたコンテンツは物理削除される |
